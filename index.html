GO 包 · 离线优先授权协议

签发一次，机器认得，
断网也算数。

license-next 把本地签名校验和周期性联网核验封装成一个包。 客户端启动时调用一次，剩下的机器码比对、有效期判断、新鲜度节流全部在内部完成—— 不需要为每条产品线重新写一遍校验逻辑。

协议 · 三态校验

每次启动，只会走到这三种结局之一

本地签名、机器码、有效期在客户端离线完成校验；只有"是否需要联网确认"是唯一分支点。 校验失败不会清空本地文件，方便排查，也防止误判导致数据丢失。

ResultOK
本地校验通过
签名、机器码、有效期、新鲜度全部通过。直接放行启动，按 3 天节流异步预刷新，不阻塞主流程。
ResultNeedRemote
仅新鲜度过期
本地文件不存在，或签发时间已超过 7 天窗口。触发一次联网校验，联网失败则拒绝启动。
ResultInvalid
明确不匹配
签名伪造、机器码不符、已过期——这是终态，不再查远端。旧文件原样保留，不做破坏性删除。
校验周期 · 两条独立时钟

硬性红线和主动同步，职责不同

新鲜度窗口是安全边界，超了必须联网；预刷新间隔只是提前感知续期或吊销的优化项，失败静默、不影响本次启动。

新鲜度窗口FreshWindow · 7 天
超窗强制联网
预刷新节流RefreshInterval · 3 天
失败静默忽略
对比 · 主流授权方案

核心校验能力对齐，运维负担更轻

跟 FlexLM、Sentinel 这类需要独立 license server 进程的老牌方案相比，license-next 只是一个 HTTP API + 本地缓存。

能力FlexLM / SentinelKeygen.sh 类 SaaSlicense-next
离线可用支持依赖网络较重7天缓存内可用
独立 license server 进程需要部署托管在SaaS不需要
多产品线统一协议格式各异支持单包统一复用
协议向后兼容扩展历史包袱重支持预留 schema_version
吊销 / 自助换绑支持支持服务端扩展项，尚未内置
集成 · main 里只有一行

调用方不需要理解周期细节

是否要联网、要不要预刷新，包内部按配置自行判断。各产品线接入时只关心成功或失败两种结果。

main.go
import license "github.com/doc-war/license-next"

checker, _ := license.NewChecker(license.Config{
  Product:  "your-product",
  PublicKey: pubKeyPEM,
  RemoteURL: "https://license.zhaoliwei.com/v1/query",
})

lic, err := checker.Check()
if err != nil {
  log.Fatal(err) // 终止核心服务
}
