license-next

轻量级客户端 License 验证框架 — 将 issuer（服务端）与 checker（客户端）分离

🧪 在线生成 CData GitHub
设计理念： client 包只依赖验签和 CData 解码；issuer 包独立，只有服务端项目才需要 import。 CKD 协议提供无状态、可逆的 License 编码，支持前向兼容。

架构

issuer (服务端)
  ├─ ECDSA P-256 签名
  ├─ CKD 派生（可选）
  └─ → LicenseSign JSON

checker (客户端)
  ├─ 本地缓存
  ├─ 签名验证
  ├─ CData 解码
  ├─ 到期/机器码校验
  └─ 远程刷新 → Fallback

快速开始

1. 生成密钥对

openssl ecparam -genkey -name prime256v1 -out key-private.pem
openssl ec -in key-private.pem -pubout -out key-public.pem

2. 服务端签发 License

package main
import "github.com/doc-war/license-next/issuer"

iss, _ := issuer.New(issuer.Config{PrivateKey: privPEM})
ls, _ := iss.Sign(&issuer.License{
    Customer:  "Acme Corp",
    ExpireAt:  time.Date(2030, 12, 31, 23, 59, 59, 0, time.UTC),
    Product:   "myapp",
    MachineID: "target-mid",
    Features:  []string{"premium"},
})
// ls.CData / ls.Timestamp / ls.Signature

3. 客户端校验

import licensenext "github.com/doc-war/license-next"

checker, _ := licensenext.New(licensenext.Config{
    Product:   "myapp",
    PublicKey: pubPEM,
    RemoteURL: "https://your-worker.workers.dev/v1/query",
})
lic, err := checker.Check(context.Background())

CData 编码

CData 是 License 合同的编码字符串，支持两种方式：

方式说明场景
base64urlJSON 序列化后 URL-safe base64 编码无 CKD 需求
CKD通过 MasterKey 派生的可逆编码，隐藏原始数据需要保密 License 内容

两种方式对客户端透明：客户端配置 MasterKey 则自动使用 CKD 解码，否则使用 base64url。

LicenseSign 结构

{
  "c_data":    "base64url 或 CKD 编码的 License",
  "timestamp": 1750000000,
  "signature": "ECDSA P-256 ASN.1 DER base64 签名",
  "revoked":   false
}

签名覆盖 CData + "|" + Timestamp，防止 CData 单独被重放。
